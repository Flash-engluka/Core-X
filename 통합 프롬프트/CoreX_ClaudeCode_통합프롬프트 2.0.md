## 0. Claude Code 시작하기 (개발 환경)

이 프로젝트는 별도 빌드 도구·프레임워크 없이 **순수 HTML + Vanilla JS + CSS**로 만듭니다. Node.js는 Claude Code 실행과 로컬 정적 서버 용도로만 필요합니다.

1. Node.js 18+ 설치 확인 (`node -v`)
2. Claude Code 설치: `npm install -g @anthropic-ai/claude-code`
   - 공식 문서: https://docs.claude.com/en/docs/claude-code/overview
3. 프로젝트 폴더에서 `claude` 실행 후 이 문서를 읽도록 지시
4. 로컬 실행/테스트: 프로젝트 폴더에서 `python3 -m http.server 8000` → 브라우저 `http://localhost:8000`
   - ⚠️ `index.html`을 파일로 직접 열면 안 됨 (Firebase SDK CDN·모듈 로드 때문). 반드시 로컬 서버 사용.

**개발 검증 원칙 (Claude Code가 지킬 것):**
- 매 단계마다 `node --check js/*.js`로 문법 검증
- 게임 로직(턴/라운드/점수)은 헤드리스 Node 스크립트로 시뮬레이션해 검증 후 진행
- 한 마일스톤이 끝나면 브라우저에서 실제 동작 확인을 사용자에게 요청

## 3. 보드 구조 (핵심)

보드는 **43칸**으로 구성된 그래프다. **이동 단위는 "정밀번호"** (1~39 + 진영 4칸 + Core).
`board_data.json`에 좌표·인접·zone 정보가 모두 들어있으니 **그 파일을 단일 진실 공급원(source of truth)으로 사용**할 것. 좌표는 원본 이미지(1414×2000) 대비 0~1 정규화 값이다.

### 3-1. 칸 종류
- **진영 홈 칸 4개**: `RUINS`, `SNOWDIN`, `WATERFALL`, `HOTLAND` (상자 보유)
- **Core 칸 1개**: `CORE` (업그레이드 전용 공간)
- **일반 칸 38개**: 정밀번호 1~39 (단, 20은 Core이므로 일반 칸에서 제외)

### 3-2. 이동 규칙

- Core에서 나올 때도 관문 4칸으로만 나갈 수 있다 (양방향).
- 진영 홈 진입: RUINS←1, HOTLAND←4, SNOWDIN←38, WATERFALL←39 (정해진 한 칸 경유).

## 4. 핵심 게임 루프 (Core Loop)

### 한 턴의 순서
```
① 이동 OR 줍기  (배타적, 1턴에 하나만)
      ↓
② (Core에 있다면) 업그레이드 실행 가능
      ↓
③ (자기 진영에 있다면) 소지품 교체 가능
      ↓
턴 종료 → 다음 플레이어
```

### 중요 규칙
- 한 턴에 최대 1회 업그레이드.
- 소지품 교체는 자기 진영에 도착한 **다음 턴부터** 가능.
- 교체 개수에 상관없이 교체 행위 자체가 1턴 소모.
- 턴 순서는 게임 시작 시 **랜덤으로 결정**.

## 6. 승리 / 라운드 구조 (확정)

- **라운드 종료 조건**: 어떤 플레이어가 **자기 진영 To-do 전체를 최초로 완료**하는 시점.
- **게임 종료(최종 우승) 조건**: **누적 100G를 최초로 달성**하는 플레이어.
- **라운드 우승 보너스 없음** (To-do 완료 +5G씩이 전부).
- **라운드 종료 시 리셋 정책**: **G(누적 점수)만 유지**, 그 외 전부 리셋
  - 인벤토리·진영 상자 비움
  - To-do 진행 초기화
  - 캐릭터 업그레이드 단계 초기화 (기본 캐릭터로)
  - 보드 위 일반 아이템 리스폰
  - 모든 말 위치를 자기 진영 홈으로 복귀
  - 다음 라운드 시작 시 전원 +15G, 조커 재배정

| 조커 | 별도 | 특수 카드 (아래 §8) |

### 7-2. 아이템 업그레이드 (Core에서)
```
Froggit     → Final Froggit
Whimsun     → Whimsalot
Lesser Dog  → Greater Dog
Moldsmal    → Moldbygg
```

### 7-3. 업그레이드권 규칙 (PDF 원문 기준)
- Core에 도착한 후 Core에서 나올 때 **즉시 사용해야 하는 업그레이드권**을 얻는다.
- Core 1회 방문 = 업그레이드권 1장.
- 업그레이드권 1장으로 **하나의 대상(플레이어 캐릭터 또는 소지품 1개)** 만 업그레이드 가능.
- **업그레이드권은 모을 수 없다** — 그 턴(또는 정해진 타이밍)에 안 쓰면 소멸.

> 예시(PDF): Undyne이 소지품 Moldsmal 2개를 든 채 Core에 다녀오면, **업그레이드권 1장으로는 Moldsmal 2개를 한 번에 못 바꾸고**, 캐릭터 본인(Undyne→Pre-Undying) 또는 Moldsmal 1개 중 하나만 가능.

### 7-4. 소지품 "교체" (상자)
- 각 진영 홈에는 **상자**가 있다.
- 자기 진영에 도착한 다음 턴부터, **인벤토리 ↔ 상자** 사이에서 아이템을 꺼내고 넣는 것이 "교체".
- 교체 수량에 상관없이 1턴 소모.

### 7-5. ⚠️ 미확정 (사용자 입력 대기) — Q3
**일반 아이템(Froggit/Whimsun/Lesser Dog/Moldsmal)이 보드의 어느 칸에 놓이는지**는 아직 미정. 사용자가 별도 그림으로 배치를 제공할 예정.
→ 그 전까지는 데이터 구조만 만들어 두고(예: `board_data.json`의 각 노드에 `items: []` 필드 추가 가능하도록), 배치 확정 시 주입할 수 있게 설계할 것. 일반 아이템 줍기 비용은 **무료**.

## 8. 조커 시스템

- 매 라운드 시작 시 **랜덤한 1명**에게 조커 1장 지급.
- 보유 시 자기 턴에 언제든 사용 가능 (1회용, 사용 후 소멸).
- **다음 라운드로 넘어가면 안 쓴 조커도 소멸** (재배정됨).
- 사용 시 셋 중 하나 선택:

| 번호 | 효과 |
|------|------|
| ① | 즉시 Core로 이동 (업그레이드권 1장 획득) |
| ② | 다른 플레이어 아이템 1개 강탈 — **사용자가 대상·아이템 지목, 강탈당한 쪽에 알림** |
| ③ | 더블 턴 (이번 턴 한 번 더 행동) |

## 13. 온라인 멀티플레이 (Firebase)

### 기술
Firebase Realtime Database (서버리스, 실시간 동기화). 4인 접속·턴 동기화·상태 공유.

### 방 시스템
- 4자리 랜덤 코드로 방 생성
- 방장이 시작 권한 보유
- 4명 준비 완료 시 캐릭터 선택 화면 자동 전환

### 데이터 구조 (예시)
```json
{
  "rooms": {
    "{roomCode}": {
      "host": "{uid}",
      "players": {
        "{uid}": {
          "name": "닉네임", "character": "Toriel",
          "gold": 0, "inventory": [], "box": [],
          "position": "RUINS", "todo": [],
          "hasJoker": false, "ready": false
        }
      },
      "gameState": {
        "phase": "waiting|char-select|playing|ended",
        "currentTurn": 0, "round": 1,
        "boardItems": {}, "logs": []
      }
    }
  }
}
```

### Firebase 설정 처리
- `js/online.js` 상단에 `FIREBASE_CONFIG` 객체를 두고, 초기값은 placeholder(`YOUR_API_KEY` 등).
- placeholder 상태에서 온라인 진입 시 "Firebase 설정 필요" 안내 모달 표시, 로컬 모드는 정상 동작.
- 사용자가 직접 Firebase 콘솔에서 프로젝트 생성 → Realtime Database 활성화 → 웹 앱 등록 → config 6~7줄을 붙여넣는다.

### 로컬 플레이 (오프라인)
- 동일 기기·화면에서 4인 턴제 핫시트.
- localStorage로 상태 저장, 새로고침 시 복구.

## 14. 파일 구조

```
corex/
├── index.html          # 모든 화면 + 모달 + 토스트
├── SPEC.md             # (이 문서)
├── board_data.json     # 보드 그래프 (단일 진실 공급원)
├── README.md           # 실행/설정 안내
├── css/
│   └── style.css       # 변수·픽셀폰트·레이아웃·애니메이션
├── js/
│   ├── main.js         # 화면 전환, 별 배경, 이벤트 와이어링, 이동 모드
│   ├── game.js         # 게임 상태/규칙 엔진 (턴·라운드·점수·To-do)
│   ├── board.js        # Canvas 보드 렌더링 + 이동 그래프 (board_data.json 로드)
│   ├── sprites.js      # 48×48 픽셀 아트 캐릭터/아이템
│   └── online.js       # Firebase 연동
└── assets/
    ├── sprites/        # 추후 캐릭터 이미지
    └── audio/          # 추후 BGM/SFX
```

### 렌더링·입력
- 보드: HTML5 Canvas 2D, 정규화 좌표 사용(어떤 크기에도 대응), 60fps requestAnimationFrame.
- UI 패널: HTML+CSS 픽셀 스타일.
- 보드 클릭: Canvas MouseEvent → 가장 가까운 노드 판별. 모바일 터치 지원.
- 반응형: PC 설계 후 모바일(≤900px) 단일 컬럼 레이아웃, Canvas 동적 리사이즈.

- **검증**: 헤드리스 Node 스크립트로 라운드 진행·점수 누적·승리 판정 시뮬레이션

### M3 — 보드 & 이동
- `board_data.json` 로드, 43칸 그래프, 노드/간선/말/하이라이트 렌더링

- Core 관문(16/17/23/24) 진입 규칙, Core 도착 시 업그레이드권 부여
- 조커 ① Core 이동 / ③ 더블 턴
- **검증**: BFS로 전 칸 연결성·진영↔Core 거리(8/8/10/10턴) 확인 + 브라우저 이동 테스트

### M4 — 아이템 & 액션 (Q3 입력 필요)
- 일반 아이템 보드 배치(사용자 그림 확정 후), 줍기/업그레이드/교체 액션
- 인벤토리·상자 UI 연동, 특수 아이템 비용 처리
- To-do 자동 진행(아이템·업그레이드 달성 시 판정), 조커 ② 강탈
- **검증**: 한 진영 To-do 전부 완료 → 라운드 종료까지 플레이
